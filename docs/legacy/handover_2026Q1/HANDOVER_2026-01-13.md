## 一、项目目标（你在解决什么问题）

建立一套 **可审稿人认可的、可复现的全景室内布局标注与质量评估流程**，核心指标体系量化两个独立维度：

1. **工作量幅度**（改了多少）：IoU_edit、boundary RMSE 等，不承诺正确性
2. **标注质量与可靠性**（改得对不对）：IAA、leave-one-out 共识、专家评分 $r_u$ 及其 bootstrap CI

技术亮点：

- **角点优先（Corner-only）**：标注者只标角点（keypoints），墙面 polygon 可选；大幅降低标注成本与歧义
- **方案 A 专家评分**：用纯人工 Manual 条件的校准集估计 $r_u$，避免"半自动初始化把大家拉齐"污染"人的能力"评估
- **严格 OOS 分流**：当 Manhattan/cuboid 假设不成立（错层/分层天花/弧形墙）或"房间边界无法唯一确定"（必须猜）时，主指标默认剔除，单列报告
- **门控齐全**：标准 layout 指标仅在角点配对、覆盖率、深度渲染等全部通过时才计算，并记录失败原因

核心研究问题：

- RQ1（效率）：半自动初始化相对纯人工的时间节省率？
- RQ2（质量）：时间节省是否以精度损失为代价？
- RQ3（分配）：能否用 $r_u$ 识别专家，并通过"难例/复核优先分配给高 $r_u$"进一步提升一致性/降低时间？

输出形态：论文式 method + experiment（先 sections 2–3.6，暂不写 intro/related work）+ 开源级工具链（evaluate 脚本 + Label Studio 配置 + SOP）。

   - 方法必须带公式定义 + 文字解释 + 适用范围/局限说明

2. **"工作量幅度 ≠ 标注质量"必须形成对比** ✓
   - 改动幅度可能是"瞎改"，需要用一致性/共识来证明改动并非噪声

## 三、过往已落地的工程改动（当前系统能力基线）

### 3.1 评估体系扩展（tools/analyze_quality.py）

从"仅图像空间粗指标"扩展为**四层指标 + 严格门控**：

**L1. 标准 layout 指标**（与 HoHoNet/HorizonNet 社区对齐）

- `layout_2d_iou`：floor polygon 投影到地面平面后的 IoU
- `layout_3d_iou`：柱体体积 IoU 近似（floor area × ceiling height）
- `layout_depth_rmse`、`layout_delta1`：**layout-rendered depth**（由角点几何渲染得到的深度）对比
- **门控条件**：cor_id 转换成功、配对覆盖率 ≥ 某阈值、角点配对结构可解析（odd/x 不一致会失败）
- **记录**：`layout_used`（true/false）、`layout_gate_reason`（normalize_failed / odd_points / low_coverage / x_inconsistent / depth_failed / out_of_scope / scope_missing / exception 等）

> 说明：`n_pairs_mismatch` **不再**作为 L1 标准 layout 指标的 gate 条件（见 P0-1 方案与后续补充交接）。点数不匹配只会影响需要 1-1 对应的 pointwise 指标（见下方 L2）。

**L2. 图像空间鲁棒指标**（不依赖深度/3D，稳定性强）

- `iou_manual`/`iou_corner`：polygon IoU（手画或角点派生）
- `pointwise_rmse_px`：点对点 Hungarian RMSE（需要配对覆盖率 ≥ 90% 且角点对数一致）
- `boundary_rmse_px`：边界曲线（上下沿）上采样后的 RMSE（最鲁棒，角点数不同时仍可用）

**L3. 可靠性指标**（多标注任务）

- `iaa_t`：任务内 pairwise IoU 中位数（**仅对 scope=In-scope 计算**）
- `consensus_uid_loo`、`iou_to_consensus_loo`：leave-one-out 共识与该标注的距离
- `iou_to_others_median`：不依赖代表选择的直接 agreement（稳健指标）
- `ru_median_iou` 及 `ru_ci_low/high`：跨任务取 iou_to_consensus_loo 的中位数 + bootstrap CI（仅 Manual 校准集、仅 In-scope）

**L4. OOS 统计与分布**

- `is_oos`：标注者选择的 `scope` 是否为 OOS 子类之一
- 单列报告：OOS 比例、OOS 子类分布（几何不成立 vs 边界不可判定 vs 错层多平面 vs 证据不足）

**门控与记录**

- CSV 新增字段记录所有门控失败原因，避免"只报能成功的样本"造成统计偏差
- 质量解析统一使用 `--quality_mode v2`（本仓库按 v2 字段口径实现与维护）

### 3.2 Label Studio 配置与字段演化

从混杂"quality 多选"改为**三字段正交设计**：

**新版字段（v2，推荐正式使用）**

- `scope`（单选，决定是否进入主指标）
  - `In-scope：只标相机房间`：能在不猜的情况下稳定闭合
  - `OOS：几何假设不成立`：弧形墙/斜顶/非 Manhattan
  - `OOS：边界不可判定`：没有清晰停止点、必须靠语义猜
  - `OOS：错层/多平面`：下沉/楼梯井/分层地顶
  - `OOS：证据不足`：质量差、关键细节缺失
- `difficulty`（多选，解释耗时/误差来源）
  - 遮挡、低纹理、拼接缝、反光、低质/被遮罩影响（例如上下被 mask/黑边导致证据缺失）
  - `residual`（尽力调整但 3D 仍不佳）：不等于 OOS，用于难例分层
- `model_issue`（仅半自动，多选）
  - 跨门扩张、漏标、角点漂移、配对异常、预标注失效

**旧版字段（legacy）**

- 已弃用且不再维护；如有早期试标数据，建议用最新 XML 重新导出或做离线迁移后再分析

**配置文件**

- `label_studio_view_config.xml`（含 model_issue）：用于 SemiAuto_Test 等项目
- `label_studio_view_config_manual.xml`（无 model_issue）：用于 Manual_Test 等项目

### 3.3 标注流程自动化

**userscript（Tampermonkey）**

- `tools/ls_userscript.js`：在 Label Studio 页面自动记录 active time（有效标注时间）到 `/log_time` endpoint

- Nginx 反向代理 + CORS 头处理，确保纹理加载成功

### 3.4 数据准备工具

**create_labelstudio_split.py**

- 从 `label_studio_import_docker.json` 任务池按 `--seed` 一次性切分互斥集合

## 五、本轮对话确定的关键决策（方法论与实验设计层面）

### 5.1 方案 A：用纯人工 Manual 校准集估计专家评分

**为什么**：

- 半自动初始化会把所有标注者"拉齐"（共识高），使得 $r_u$ 反映的不再是"人的能力"，而是"工具的拉齐效果"
- 审稿人会问："你怎么证明 $r_u$ 区分不出来是人的差异 vs 工具差异？"

**怎么做**：

- 所有参与者在 `calibration_manual` 项目中完成 30 张图（每张 k≥3 人标注）**纯人工**
- 计算 $r_u = \mathrm{median}_t \mathrm{IoU}(A_{t,u}, C_t^{(-u)})$（LOO 共识），带 bootstrap CI
- 用 $r_u$ 进行验证集的任务分配（难例/复核优先给高 $r_u$）

**保证**：

- 避免 circularity（自我参与共识）：用 LOO 共识 $C_t^{(-u)}$（不包含用户 u 自身）
- 避免工具污染：来自 Manual 条件，不混 Semi
- 统计有效性：至少 5 个多标注任务（≥ 3 人）才报告 $r_u$

### 5.2 IAA 与 $r_u$ 只在 In-scope 上计算

**原理**：

- OOS 定义为"任务目标不可复现或假设不成立"，强行纳入 IAA 会把"目标不唯一导致的方差"混进"人的稳定性"
- 结果：IAA 虚低，$r_u$ 估计有偏

**操作**：

- CSV 记录时：给每个 (task, user) 标记 `is_oos`
- 计算 IAA_t 时：筛选 scope=In-scope 的样本后再取 pairwise IoU 中位数
- 计算 r_u 时：只用 Manual 校准集里 scope=In-scope 且多标注的任务
- 单列报告：OOS 比例、OOS 子类分布，以及（可选）OOS 判定本身的标注一致性

### 5.3 门洞/开放式连通规则（写死以避免"需要理解算法"）

**规则**：

- 若门框/墙垛清晰：边界 **止于门框处**，不跨门洞，仍为 `scope=In-scope`
- 若没有清晰停止点、必须靠语义"猜"才能闭合：选 `scope=OOS：边界不可判定`

**好处**：

- 标注员无需理解 Manhattan 假设的技术细节，只需遵守几何规则
- 可复现：审稿人可按同一规则复标 + 对比

## 九、注意事项（审稿人会卡的点）

### 9.1 专家评分 $r_u$ 的严谨性

**禁忌**

- ❌ 不要用 Semi 条件直接算 $r_u$（工具污染）
- ❌ 不要在共识中自我参与（circularity），会导致 $r_u$ 虚高
- ❌ 不要把 OOS 混进 r_u 计算（目标不可复现会污染稳定性估计）

**正确做法**

- ✓ $r_u$ 仅来自 `calibration_manual`（Manual 条件）
- ✓ 用 LOO 共识 $C_t^{(-u)}$（不包含用户自身）
- ✓ 只用 scope=In-scope 且多标注的任务
- ✓ 报告 bootstrap CI 与有效样本量（≥ 5 个多标注任务）

### 9.2 OOS 处理的透明度

**禁忌**

- ❌ 不要悄悄剔除 OOS 后再报告 IAA/r_u，让审稿人以为是全样本
- ❌ 不要"挑反例"来定义 OOS，应该事前写死规则

**正确做法**

- ✓ 明确说"主 layout 指标对 OOS 默认剔除"，并报告 OOS 比例/子类分布
- ✓ OOS 抽样规则预先固定（分层规则，不是事后挑选）
- ✓ 单列报告 OOS 的判定一致性（label agreement）

### 9.3 校准集多标注的充分性

**禁忌**

- ❌ 只有 1–2 人标注 calibration_manual（r_u 样本不足）
- ❌ 校准集和主对照有重叠（一致性相关性会升高）

**正确做法**

- ✓ 校准集每张至少 k=3 人标注
- ✓ 校准集与主对照/验证集默认互斥
- ✓ 每位参与者在校准集上至少 5–10 个多标注任务（否则 $r_u$ CI 太宽）

### 9.4 验证集的实验设计

**禁忌**

- ❌ 分配策略对照中"Baseline 随机分配，Proposed 按 $r_u$ 分配"，但两者的样本难度不一致（难度混淆）
- ❌ Validation 全是单标注，无法直接比较 $IAA_t$ 的变化

**正确做法**

- ✓ 两种分配策略使用同一批验证集图
- ✓ 采用分层随机保证难度 proxy 分布一致
- ✓ 若需要比较 $IAA_t$，建议对验证集内至少 20 张图做 2–3 人复核

### 9.5 质量 choices 的版本管理

**禁忌**

- ❌ 在正式实验中混用 v2 与旧版（已弃用）choice 文本
- ❌ 导出时不清楚 choices 的来源版本

**正确做法**

- ✓ 正式标注统一使用 v2 配置（新三字段）
- ✓ 若包含旧导出，先离线迁移/重导出为 v2 字段；否则不纳入主分析与可靠性结论
- ✓ analyze_quality.py 用 `--quality_mode v2`，且记录版本信息

### 9.6 工作量与质量的独立性

**禁忌**

- ❌ 用"改动幅度很大"直接推断"质量很好"（很可能是瞎改）
- ❌ 结论中混淆"这个标注员改得多"和"改得对"

**正确做法**

- ✓ 论文中明确分开两个维度（RQ1 效率 vs RQ2 质量）
- ✓ 给出正反例：改动很大但 IAA 很低（反例），以及改动很小但 IAA 很高（好例）
- ✓ 用一致性/共识来支撑"改动并非噪声"

## 十一、联系与问题排查

### 常见问题

**Q1：为什么 active_time 都是 0？**

- A：Tampermonkey 脚本没在标注页运行，或 token 不一致，或 /log_time endpoint 不可写。检查 README_DEVELOPER.md 的日志服务配置。

**Q2：为什么 OOS 样本的 layout_used 仍然为 true？**

- A：analyze_quality.py 版本过旧，或 --quality_mode 不是 v2。升级脚本或改参数。

**Q3：我的 r_u CI 很宽，怎么解释？**

- A：可能是样本不足（<5 个多标注任务）、或标注者一致性本身很差。检查该用户的 iou_to_consensus_loo 分布。

**Q4：门洞后有明显相邻房间，应该选什么？**

- A：如果门框清晰，止于门框即可，选 In-scope；如果没有清晰停止点必须猜，选 OOS：边界不可判定。
