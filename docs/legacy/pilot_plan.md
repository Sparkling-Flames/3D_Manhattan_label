# Pilot 试运行方案（招募 10+ 标注员前）

目标：用极小成本验证 **(1) 标注口径是否可执行**、**(2) 导出/解析是否稳定**、**(3) active time 是否可用**、**(4) OOS 分流与一致性/ru 计算是否按预期门控**。

---

## 1. 建议配置（低成本可落地）

- 标注员：2–3 人（建议至少 1 名“熟练者”+ 1 名“新手”）
- 任务量：
  - 主试跑（单标注）：每人 10–15 张（Manual 或 Semi 二选一，先优先 Semi 验证预标注工作流）
  - 校准重叠：额外 6–8 张 **同一批图**，每张由 2–3 人标注（用于快速算 IAA 与试算 $r_u$）
- 总标注量目标：约 30–45 份提交（1–2 小时内能跑完，且足够暴露问题）

> 核心思想：先把“系统链路”跑通，再谈规模化统计。

---

## 2. Pilot 数据构成建议

- In-scope：至少 60%（保证能看到正常样本的效率与一致性）
- OOS：至少 20%（保证能验证 OOS 分流、门控与统计口径）
- 难例：至少 20%（遮挡/低纹理/拼接缝/反光等，确保 difficulty 标签会被用到）

---

## 3. 验收清单（可量化 pass/fail）

### 3.1 标注完整性

- `scope` 填写率：≥ 95%（任何漏填都应当视为 UI/培训问题）
- `difficulty`/`model_issue` 使用率：不设硬阈值，但应出现“合理的非零计数”，且文本与场景匹配

### 3.2 门洞/边界规则是否可执行

抽样复核 5–10 个“门洞/开放式连通”样本：

- 门框清晰：多数应选 `scope=In-scope`（边界止于门框）
- 无可靠停止点：多数应选 `scope=OOS：边界不可判定`

若出现系统性分歧：优先改 SOP/示例图，而不是要求标注员“理解算法”。

### 3.3 Active time 日志可用性

- active time 非零覆盖率：≥ 90% 的任务能在 CSV 中看到合理的 `active_time`
- 若覆盖率低：先排查 userscript 是否只在标注页运行、token 是否一致、/log_time 是否可写

### 3.4 分析脚本输出健康度

用导出跑：

- `python tools/analyze_quality.py <export.json> --active-logs active_logs --output_dir analysis_results --metric corner --quality_mode v2`

检查：

- OOS 样本：`layout_used` 应为 false 或 `layout_gate_reason=out_of_scope`
- In-scope 样本：`layout_used` 有一定比例为 true（不是全失败）
- CSV 中 `scope/difficulty/model_issue` 列非空且可读

### 3.5 快速一致性与 ru 试算（只要能跑通即可）

对“校准重叠”那 6–8 张：

- IAA 能算出来（至少部分任务有 >1 标注）
- $r_u$ 能产出（哪怕 CI 很宽也没关系），用于验证流程而非结论

---

## 4. Pilot 结束后的迭代节奏

- 第 1 轮（当天）：只修“阻断性问题”（无法提交、字段不清、导出解析失败、active time 全 0）
- 第 2 轮（次日）：补“示例与边界案例图”（门洞/开放式连通/错层），让口径更一致
- 第 3 轮：才开始扩到 10+ 标注员，并冻结 XML 文本与分析口径
