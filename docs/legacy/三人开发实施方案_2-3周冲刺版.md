## 2. legacy交接文档吸收结果（保留 vs 过时）

已阅读：`HANDOVER_2026-01-13/16/18/21/25/26.md` 与 `AI_HANDOVER_2026Feb.md`。

### 2.1 仍应保留（与当前提纲一致）
1. **Single Source of Truth**：上游 `analyze_quality.py` 为权威计算，下游 notebook 不重算核心字段。  
2. **门控透明化**：保留 `layout_used/layout_gate_reason`、`reliability_used/reliability_gate_reason`。  
3. **scope/OOS 口径**：In-scope 与 OOS 分流，OOS 不混入主估计。  
4. **active_time 稳定采集**：`ls_userscript.js` + `cors_server.py` 方案有效。  
5. **P0-1 修复思想**：避免把高难样本静默剔除，保持可审计覆盖。

### 2.2 需谨慎使用（部分过时/与当前提纲不完全一致）
1. legacy 中一些“以旧分组名为中心”的流程，需以你当前提纲章节中的分组定义为准。  
2. 部分文档把 `Calibration_semi` 作为固定主流程组；当前提纲主线更强调 `Calibration_manual` 作为可靠度核心锚点。  
3. 一些早期 notebook 输出与现提纲第4章表格结构（A/B/B2/C/D）不一，需要按当前审计结构重排。  
4. `AI_HANDOVER_2026Feb` 中分布审计愿景较大（Embedding/MMD 等），冲刺期仅保留最小可执行版本，不做全量扩展。

结论：legacy 可作为工程稳健性参考，但**论文与执行口径必须回到你当前 `sections/` 提纲**。

## 10. 风险与压缩策略

| 风险 | 触发信号 | 压缩策略 |
|---|---|---|
| PreScreen 回收慢 | 第2周中期回收<50% | 缩小首轮PreScreen任务量，先形成准入核心池 |
| 工具改造拖延 | P0改动未在第1周完成 | 走手工流程+固定模板，先保实验推进 |
| 计时质量不稳 | active_time异常率高 | 先用 lead_time 做兜底，并单列审计解释 |
| 审计输出不齐 | 第3周无法导出最小包 | 降级为CSV模板输出，图形后置 |
