## 1. 与论文提纲的一致性核查

### 1.1 已对齐项（与提纲一致）

- 标注执行链：`prepare_labelstudio_docker.py` + `create_labelstudio_split.py` + Label Studio SOP。
- 质量分析链：`analyze_quality.py` + `aggregate_analysis.py` + `visualize_output.ipynb`。

- 论文协议基础：Pilot→PreScreen→Main、Trap 隔离、口径分层（T/I/M）与提纲 2.0/3.1/4.1 一致。
- 数据口径基础：`test` vs `test_no_occ` 的主报告/敏感性分析口径与提纲 3.1 已锁定。

### 1.2 偏差项（与提纲仍有出入）

- **出入B：non-IID/OOD链路未形成稳定工具**。`noniid_split_generator.py` 维护性差，不满足“预注册+可审计”要求。
- **出入C：RQ3 的“路由策略输出”缺最后一跳**。缺直接生成任务分配清单的脚本。
- **出入D：实验先行节奏未显式化**。需把“先开展实验、后完善自动化”写入项目执行规则。

## 2. 总体执行原则（导师语音映射）

### 2.1 实验先行（优先级最高）

- 第一优先：按提纲跑通 Pilot → PreScreen → Main 的真实任务流。
- 第二优先：完成 RQ1/RQ2/RQ3 的最小可发表证据闭环（主终点 + CI + 审计产物）。
- 第三优先：在同一流程内补 non-IID/OOD 分层结果，不额外扩展无关功能。

### 2.2 自动化并行（不阻塞实验）

- 统一流程入口：串联“拆分→导入准备→分析→汇总→报告”。
- 统一配置：单一 `workflow_config.yaml` 锁定 split、阈值、路径、seed、口径。
- 统一审计产物：每一步落盘 manifest（输入、参数、输出、时间戳、执行角色）。

## 3. 推荐开发路线（4周，实验驱动）

## Phase A（第1周）：实验启动周（允许半手动）

### 目标

- 不追求全自动，优先保证第一轮真实实验可执行、可追溯。

### 交付物

- `docs/experiment_runbook_v1.md`（一步一命令 + 失败回滚）
- `tools/workflow_manifest.py`（最小版，记录每步输入/输出）
- 任务包级别审计清单（Pilot/PreScreen/Main 的 task_id 隔离记录）

### 验收标准

- 3名执行者按 runbook 可在半天内完成一轮“导入→标注→导出→分析”。
- Trap 与 Main 物理隔离记录完整。

---

## Phase B（第2-3周）：提纲核心结果周（RQ1/2/3）

### 目标

- 产出可写入论文正文的主结果：效率、一致性、路由收益、过程证据。

### 交付物

- `analysis_results/quality_report_*.csv`、`reliability_report_*.csv`（完整口径）
- Worker 四象限图、Worker×Scene 矩阵、反例案例库初版
- Scope 冲突矩阵与高频 tag 冲突统计表（用于不确定性定位）
- Random/Global/Stratified 三策略离线比较表

### 验收标准

- 每个 RQ 至少 1 个预注册主终点完成统计检验与效应量报告。
- non-IID/OOD 分层结果可复现实验命令化。

---

## Phase C（第4周）：自动化收口周（不改研究问题）

### 目标

- 在不改变实验口径前提下，完成 workflow 脚本化与投稿材料归档。

### 交付物

- `tools/run_workflow.py`（统一CLI入口，支持 `pilot|prescreen|main|report` 子命令）
- `docs/reproducibility_pack.md`（复现实验打包清单）
- 图表自动产出脚本（worker图、OOD分层图、策略对比图）

### 验收标准

- 一条命令可复现实验主表和主图。
- 任一结果都能追溯到输入数据、参数版本、执行日志。

---

## 4. 三人开发小组任务分配（建议）

## 角色A：Workflow/平台负责人

### 核心职责

- 统筹 protocol 与入口脚本，保证实验先跑通、全流程可审计。

### 负责模块

- `run_workflow.py`（总编排）
- `workflow_manifest.py`（审计产物）
- `experiment_runbook_v1.md`（操作手册）
- split 与数据清单规范（task_id/title/version 管控）

### KPI

- 全流程一次通过率、复现时间、审计清单完整率。

---

## 角色B：算法/策略负责人（non-IID + consensus + routing）

### 核心职责

- 把导师强调的“理论高度”落到可检验策略：分布shift、工人分类、加权共识、任务路由。

### 负责模块

- non-IID/OOD 切分与敏感性实验
- 共识与路由策略评估（Random/Global/Stratified）
- 过程性证据生成（Worker画像、Scene矩阵）
- 混淆/冲突审计输出（Scope 冲突矩阵、tag 冲突统计）

### KPI

- 策略对比显著性；shift 检测稳定性；路由收益（一致性/失败率/成本）。

---

## 角色C：数据/实验运营负责人（标注组织 + 质量闭环）

### 核心职责

- 把 Label Studio 任务流和人力组织跑起来，保证数据质量与节奏。

### 负责模块

- 标注员培训与质检（scope/OOS一致口径）
- active_logs 监控与异常排查
- 反例库维护与专家复核台账
- 导出、归档、命名规范执行

### KPI

- 字段完整率；mixed-scope 比例；日志覆盖率；多标注任务完成率。

---

## 5. 协作机制（避免三人互相阻塞）

- 每周固定两次同步：
  - 周二：技术同步（接口/字段/风险）
  - 周五：实验复盘（指标与失败案例）
- 接口契约先行：
  - 先冻结 CSV schema 与 manifest schema，再开发模块。
- 时间盒策略：
  - 每周至少完成一轮真实数据实验，自动化开发不得阻塞实验进度。
- 分支策略：
  - `main` 只放可跑版本；功能开发走 `feature/*`；每个 PR 必须附最小复现实验命令。

1. 完成一次真实 Pilot（3人、固定任务包、完整导出与分析）。
2. 新增 `workflow_manifest.py`：记录每步输入输出与参数。

4. 固化 `workflow_config.yaml`：统一路径/seed/阈值/口径。
5. 输出“提纲主终点对照表”：每个 RQ 对应命令、CSV 字段、统计脚本。

## 7. 论文定位（与当前提纲一致）

- 本文定位：**“protocol + quality control + distribution-aware routing”**。
- 不承诺完整在线自演化重训练闭环；重训练保留为未来工作。
- 先做到“瑕不掩瑜”：
  - 工具可用
  - 协议可审计
  - 过程证据完整
  - 在 non-IID 场景下有清晰收益

---

## 8. 风险与应对

- 风险1：标注窗口短、人员波动大
  - 应对：优先做 PreScreen + 核心子集，保证论文主结论样本先闭合。
- 风险2：过度追求自动化导致延期
  - 应对：先人工可执行，再把高频步骤自动化。
- 风险3：口径漂移（test/test_no_occ、字段映射）
  - 应对：在 config + manifest 里写死，所有报告自动带口径说明。
