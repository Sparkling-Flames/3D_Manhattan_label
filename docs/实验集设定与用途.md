# 实验集设定与用途

本文当前采用三类前置样本池：manual 几何锚点池、semi 纠错/盲信池、OOS gate 池。三者角色不同，不应混用。

## 1. PreScreen_manual

### 1.1 manual 锚点集

- 目标规模：20 到 22 个。
- 所属总池：
  - 当前设想为 PreScreen_manual 总共 30 个，其中 20 到 22 个为人工锚点，剩余 8 到 10 个为非锚点随机样本。
- 选择原则：
  - 人工确保覆盖关键场景和关键难度层。
  - 只纳入可稳定给出几何专家参考的 In-scope 样本，不把 OOS/证据不足样本混入几何 GT 锚点。
  - 目标是让 prescreen 真能看到不同 failure family，而不是全是容易样本。
- 当前优先覆盖的 manual 家族：
  - 非常简单：作为低难度基准。
  - 遮挡明显但仍可闭合：检验在遮挡下的基本稳定性。
  - 拼接缝及拉伸：检验 seam/stretch 诱导的局部折点错误。
  - 玻璃/反光：仅在专家能稳定给出参考时纳入；高歧义样本只保留为待复核或审计样本，不强行作为硬锚点。

### 1.2 manual 非锚点集

- 规模：8 到 10 个。
- 采样方式：从定义好的候选池里随机抽取。
- 目的：保留真实 base-rate，避免整个 prescreen 变成人工拼装的 showcase。
- 作用：
  - 让 admission screening 不只建立在“精心挑选的典型题”上。
  - 用于观察工人在较自然分布上的稳定性，但不承担主要阈值拟合职责。

## 2. PreScreen_semi

### 2.1 semi 锚点集

- 目标规模：18 个左右。
- 角色：
  - 不承担 manual 几何基准作用。
  - 主要用于测量纠错能力、盲信初始化风险、以及典型失败家族上的修正能力。
- 当前推荐结构：
  - 6 个“模型标注质量好”的正常初始化对照样本。
  - 12 个误导性初始化样本，覆盖 4 个左右失败家族，每类约 3 个；若某类自然失败不足，则由规则化扰动补足。
- 当前 semi 自然失败家族优先级：
  - 过度解析
  - 角点错位
  - 角点重复
  - 跨门扩张
  - 模型预标注失败
- 备注：
  - 当前 trap bank 中“漏标”“拓扑崩溃”尚未形成稳定自然案例，不应被设为 prescreen 必须覆盖家族；若后续补齐，可作为扩展家族加入。

## 3. OOS gate 池

### 3.1 用途

- OOS gate 池不进入 manual 几何 GT 锚点。
- 其职责是检验工人是否能正确拒绝继续几何标注，或正确识别“当前证据不足/几何假设不成立”。

### 3.2 当前优先覆盖的 OOS 家族

- 边界不可判定
- 证据不足
- 错层/天花板下凸
- 几何假设不成立（如弧形墙）

### 3.3 使用原则

- OOS gate 的判断正确性应与 manual 几何正确性分开计分。
- 它是 scope/gate 能力测试，不是几何精标能力测试。
- 因此不能把 OOS gate 样本混写成 manual 锚点数量的一部分。

## 4. 当前最适合的总结构

- PreScreen_manual：30 个
  - 20 到 22 个 manual 几何锚点
  - 8 到 10 个随机非锚点
- PreScreen_semi：18 个左右
  - 正常初始化对照 + 误导性初始化 trap
- OOS gate：独立小池
  - 不与 manual 几何锚点混池

## 5. 这样设计的原因

- manual 锚点负责测“基本人工几何能力”。
- semi 锚点负责测“面对模型初始化时是否会盲信、是否会纠错”。
- OOS gate 负责测“是否能拒绝不该继续做几何闭合的样本”。

如果把三者混成一个 prescreen 集合，审稿时最容易出现的问题是：

- 你到底在测工人的几何能力，还是在测其纠错能力？
- 你到底在测 In-scope 标注质量，还是在测 OOS 拒绝能力？
- 你最后得到的 admission threshold 具体对应哪一种 estimand？

因此，当前版本应坚持三池分离、角色分离、计分分离。

## 6. 总体流程

结合当前论文提纲，整套实验流程更准确的写法不是把 PreScreen、Calibration、Test、Validation 当作四个并列阶段，而是：

- Pilot：流程验证与协议边界冻结，不进入主结论。
- PreScreen：建立独立真值锚点、完成 admission、锁定加权相关阈值。
- Calibration：在通过 PreScreen 的工人池上估计正式能力统计量，并冻结后续路由所依赖的参数。
- Main：正式执行阶段，内部再拆成 Test 与 Validation 两个不同用途的执行面。

换句话说，真正的主线应理解为：

- Pilot
- PreScreen
- Calibration
- Main(Test + Validation)

其中 Test 与 Validation 不是和 PreScreen、Calibration 同一级别的“前置阶段”，而是 Main 阶段里服务于不同研究问题的两个运行面。

## 7. 各阶段的职责

### 7.1 Pilot

- 目的：验证任务创建、采集协议、扰动算子和审计字段是否可运行。
- 锁定内容：
  - manual / semi / OOS 三池边界。
  - 同一 base_task_id 不跨池复用。
  - semi trap schema。
  - 扰动算子集合、强度档位、随机种子协议。
  - 候选权重网格与 split 协议。
  - 审计字段与日志 schema。
- 不锁定内容：最终 admission 数值、最终 routing 阈值、最终 scene-routing 参数。

### 7.2 PreScreen

- 目的不是得到最终 worker profile，而是先建立一个独立、可审计的准入与初始锚定过程。
- PreScreen_manual 的作用：
  - 提供带专家参考标准的 manual anchors。
  - 估计初始可靠度锚点 r_u^(0)。
  - 锁定 admission 与 w_max。
  - 识别明显不合格或工程不可用的工人。
- PreScreen_semi 的作用：
  - 测量纠错能力。
  - 测量盲信初始化风险 T_u。
  - 识别面对误导性初始化时是否会“几乎不改就提交”。
- OOS gate 的作用：
  - 测 scope / gate 能力。
  - 不与几何精标能力混算。

因此，PreScreen 的核心产物是：

- admission 结果
- 初始可靠度锚点 r_u^(0)
- 盲信初始化风险的早期证据
- 加权共识的独立权重锚点与 w_max

而不是最终的 r_u、r_u^(s) 或完整 routing 画像。

### 7.3 Calibration

- 这是正式估计 worker 能力统计量的阶段，而不是“再做一轮 prescreen”。
- Calibration_manual 的职责：
  - 估计全局可靠度 r_u。
  - 估计场景特异可靠度 r_u^(s)。
  - 估计其置信区间与 LCB(r_u)。
  - 识别条件性脆弱风险 C_u。
  - 冻结核心场景集合和 scene-routing 启用规则。
  - 构建 d_t 的 calibration-only embedding 参考库，并据此冻结 tau_d 与 I_t^{OOD}。
- Calibration_semi 的职责：
  - 只服务 RQ2 条件对照。
  - 不用于估计 r_u 或 r_u^(s)。

因此，Calibration 是从“PreScreen 的独立粗锚点”过渡到“可部署的正式 worker statistics”的关键阶段。

### 7.4 Main

Main 阶段应拆开理解为两个不同的执行面：

- Test：主要服务 RQ1。
- Validation：主要服务 RQ3。

#### Test

- 关注 semi 初始化是否节省时间、是否改变单次标注行为。
- 更偏条件效应与最终表现。
- 不是 routing 的主证据场。

#### Validation

- 关注 routing / stress mode / OOD-aware 触发是否真正改善一致性代理与预算效率。
- 这里才是 d_t / I_t^{OOD} 真正发挥路由作用的主舞台。
- Validation_OOD 与 Hard 子集也都在这个执行面中定义与报告。

## 8. Calibration 那 12 张全员必须标注的图，到底是什么

当前提纲中，这 12 张更准确的角色不是“专家真值锚点”，而是“全员公共比较子集”。

它们的职责是：

- 提高 worker 间可比性。
- 在进入 Calibration_core 之前提供共同的比较基线。
- 稳定 LOO 共识，不让 worker 间比较完全依赖不同题目。

它们不承担的职责是：

- 打破加权共识循环论证。
- 作为新的专家真值锚点去重新定义 w_u。

避免循环论证的那个独立锚点，已经由 PreScreen_manual 的 20 到 22 个专家参考样本承担。也正因为如此，Calibration 的这 12 张不应再被理解成另一套专家真值锚点，否则会把 PreScreen 与 Calibration 的分工重新混淆。

所以更严谨的理解是：

- PreScreen_manual anchors：独立真值锚点。
- Calibration 12 张：全员公共比较子集。

从命名上看，论文里虽然仍保留 Calibration_anchor 这个名字，但在解释层面必须始终强调：

- 它是 common-item anchor。
- 不是 expert-reference anchor。

## 9. 当前版本下，三阶段流程还需不需要微调

需要微调，但不是推翻重做。

### 9.1 需要微调的点

- 第一，明确 PreScreen 的产出边界。
  - PreScreen 负责 admission、r_u^(0)、T_u 和 w_max。
  - 它不负责给出最终 worker profile。
- 第二，明确 d_t 的正式生效时点。
  - d_t 虽然是“对任务标注前可得”的代理信号，但它依赖 Calibration 构建的参考库。
  - 因此，对 Main 阶段任务而言它是标注前可得；对整条研究流水线而言，它不是在 PreScreen 之前就正式可用。
- 第三，明确 Calibration 12 张的命名与角色。
  - 它们应被解释为全员公共比较子集，而非专家真值锚点。

### 9.2 不需要推翻的点

- 不需要取消 PreScreen_manual 的专家锚点。
- 不需要把 Calibration 再变成“专家复标阶段”。
- 不需要把 Test 和 Validation 合并。

### 9.3 当前最稳的统一口径

如果以后需要在文中、答辩或 rebuttal 里用一句话概括，建议固定使用下面这套说法：

- PreScreen：独立真值锚定 + admission。
- Calibration：正式能力估计 + 参考库冻结。
- Main：
  - Test 负责 RQ1。
  - Validation 负责 RQ3。

这样最容易避免把“准入”“正式估计”“部署验证”三种不同 estimand 混在一起。
