## 二、C1 / C2 既有评判回顾

此前结论：方向正确，但存在以下关键问题。

### 2.3 新增：PreScreen trap 双来源与自然失败稀缺处理

针对本轮提出的三类真实失败样本（典型跨门扩张、过度解析、整体失效），新增结论如下。

1. 当前论文提纲已经具备“双来源”雏形，但属于“不对称双来源”：主来源是规则化扰动算子，自然失败来源是补充生态效度来源。若要把真实失败样本正式纳入 `PreScreen_semi` trap，需要把正文明确写成“semi 盲信子集中的固定小配额 natural-failure trap bank”，而不能再只写“附录稳健性披露”。
2. 同一图像不得同时进入 `PreScreen_manual` 与 `PreScreen_semi`。这不是实现偏好，而是防记忆污染与暴露泄漏的基本要求；更严格地说，同一 `base_task_id` 不得跨池复用，即便初始化版本不同也不行。
3. 需要区分 manual 与 semi 的约束重点。manual 专家锚点的核心要求是覆盖主要 difficulty strata，以评估不依赖初始化的基础标注能力；它不需要镜像 semi 的失败类型配额。相反，planned / realized quota 的披露要求主要针对 semi 的 trap 组成，而不是 manual 锚点池。
4. 当 semi 侧自然失败样本不足时，最严谨的做法不是把同图复用到 manual，也不是事后挑最难的少量例子硬凑，而是：
	- 预注册一个 fixed natural-failure quota；
	- 若某类型真实失败样本不足，则下调该类型 realized quota；
	- 用同类型的规则化扰动样本补足 `PreScreen_semi` trap 总配额；
	- 同时披露 planned vs realized quotas，避免 cherry-picking。
5. 若需要保留这三张图作为 semi trap，它们应被专家精标并记为 `source_type = natural_failure`；manual 侧应改用不重叠图像池中的专家锚点样本，而不是复用这三张图。
6. 三个案例的主标签边界建议一并写死到执行文档与 XML hint：
	- 墙面凸起/圆柱被识别成真实拐角：优先 `over_parsing`；
	- 整体几乎完全错位、基本需从零重画：优先 `fail`，除非自动合法性检查明确表明其已构成非法拓扑；
	- 把门后相邻空间卷入当前 camera room：优先 `overextend_adjacent`。

### 2.4 新增：对当前 `perturbation_operators.md` 算法设计的审稿式评判

就“semi 部分可收集少量自然失败图，其余通过算法构造”这一方向本身，我认为是合理的；问题不在总方向，而在当前算法规格写得仍然过于工程直觉化，尚不足以支撑严苛审稿。

优点：
1. 已经明确 semi 的目的不是恢复真实门洞语义，而是生成可修复、可审计的误导性初始化；这点方向正确。
2. 已把 `overextend_adjacent`、`corner_drift`、`corner_duplicate`、`topology_failure`、`fail` 区分为不同 operator family，便于冻结与复现。
3. 已有 seed、freeze plan、边界条件表，说明作者意识到了可复现性要求。

主要问题：
1. `OverextendOperator` 目前“随机选一条边、推终点、再 clamp”的实现过于粗糙。它可以制造形变，但未必稳定对应“跨门扩张”这一具体失效家族。对审稿人而言，这更像 generic polygon inflation，而不是 `overextend_adjacent` 的可信 proxy。
2. `CornerShiftOperator` 当前写成按强度随机移动 1--3 个角点、最高 30px，这不够贴近你现在收紧后的定义。`corner_drift` 更合理的原型应是一个或少量局部边界的近邻错位，且拓扑关系基本不变；当前写法容易把 drift 做成半个 `fail` 或 `topology_failure`。
3. `OverParsingOperator` 目前只写成在任意相邻边间插点。这能制造“点变多”，但未必对应“柱体/墙面凸起/家具边缘被误判为真实角点”。对这类语义依赖强的失败家族，natural exemplar bank 往往比纯几何插点更有说服力。
4. `FailOperator` 返回空列表虽然方便，但如果没有写清 userscript、3D 预览和质量分析如何消费，就会被质疑是在把 intentional invalid 混入正常初始化接口。
5. 全文仍以像素平面+clamp 思维描述越界处理，这与 panorama 的周期几何不符，也与现有实现中的百分比域真源不符。

更严谨的方法建议：
1. 采用“家族分治”而不是“所有失效都尽量算法化”的策略。
	- 几何原生、语义依赖弱的家族：`corner_drift`、`corner_duplicate`、`underextend`，以 synthetic operators 为主；
	- 语义依赖强的家族：`overextend_adjacent`、典型 `over_parsing`，优先使用 natural-failure exemplar bank，小样本不足时再用弱化的几何 surrogate 补足；
	- `topology_failure` 与 `fail` 维持 intentional-invalid 小配额。
2. `corner_drift` 应改成“局部、近邻、拓扑保持”的 operator：优先只扰动一个物理 corner boundary 或一对局部上下边界，幅度用归一化/角度比例锁定，而不是随图像分辨率硬编码像素位移。
3. `overextend_adjacent` 若保留 synthetic 版本，应避免声称自己在模拟真实门框检测；更稳妥的写法是：仅在冻结清单中对专家预先批准的可扩张样本/边索引施加 outward extension，作为 `overextend_adjacent` 的几何 surrogate，而不是对任意边随机延长。
4. `over_parsing` 的 synthetic 版本应降级为“弱 surrogate”，主要作用是制造额外伪折点；真正代表柱体/墙凸等案例的展示与盲信考察，应更多依赖 natural exemplar bank。
5. 统一表示层：冻结清单与审计侧以归一化/百分比域为真源，运行时再映射到像素；横向规则明确为 `x wrap`，纵向规则为 `y clamp`。

## 三、对当前四份约束的逐项评判

### 3.1 `merged_all.md`

#### 优点
- 已明确 `merged_all.csv` 是 Single Source of Truth，这与 Phase 1 分工和论文审计链是一致的。
- 已覆盖主要模块所需字段：元标签、质量指标、可靠度、场景、`d_t`、四类反例。
- 已补入 Type 4 相关审计字段，这一点是必要的。

#### 主要问题
1. `scope` 口径不够严谨：文中示例既出现 `in_scope`，而 XML alias 实际是 `normal`。如果发布层想用 `in_scope` 作为 canonical 值，需要明确写出 canonicalization 规则；否则会和 `tools/analyze_quality.py` 的 alias 映射口径打架。
2. 缺少 `worker_group_reason` 与 `group_rule_version`：论文正文已锁定这两个字段，作为导出与图 D 的可追溯字段，不能只留 `worker_group`。
3. `d_t` 字段定义过于简化：如果 `merged_all.csv` 是 SSOT，那么按 C2 至少还应有 `d_t_status`、`d_t_ref_hash`、`d_t_model_ver`、`d_t_failure_reason`。
4. `type1_flag`–`type4_flag` 的示例规则写得太像最终定值，和论文“预注册规则/自动筛选/专家复核”的表述相比，过于简化。建议改成“示例性解释”，不要在格式文档里固化可能漂移的判据数值。

#### 结论
- 适合做字段总表；
- 不适合承载最终反例规则或 canonical alias 映射，除非再补版本化说明。

### 3.2 `visualize_output_v2.md`

#### 优点
- 正确地把 notebook 定位为“读取 `merged_all.csv` 的下游”，没有误写成数据源。
- 已覆盖 Cell 组 1–5、图 D、表 C 的主要输入字段。
- `active_time` 审计部分写得相对完整，能支撑 RQ1 的基本展示。

#### 主要问题
1. `condition` 大小写与主表不一致：这里写的是 `Manual/Semi`，而 Phase 1 和其他约束常写 `manual/semi`。这种细节在 notebook 中最容易引发静默过滤 bug。
2. 图 D 仍缺少 `worker_group_reason` / `group_rule_version` 的可视化输入说明；而论文正文已明确 Vulnerable 的形状编码应与 reason 对齐。
3. 表 C 说明不够：缺少 `N_{u,s,min}`、CI、局部失效三条规则、启用率/退化率披露等与 04 报告对应的关键元素。
4. `active_time` 审计说明仍偏 notebook 经验性总结，尚未完全对齐 04 报告要求的 unknown task/annotator/project、script_version 缺失、多 session 占比等固定披露项。

#### 结论
- 足以指导 Dev B 出图；
- 但还不算“论文级绘图契约”，因为少了与 04 章节一一对应的强制披露项。

### 3.3 `perturbation_operators.md`

#### 优点
- 最大优点是细；确实已经把“每个算子做什么”展开到了接近函数说明的程度。
- 明确区分了 `AcceptableOperator`、`OverextendOperator`、`CornerShiftOperator` 等，这对实现者友好。

#### 主要问题（本轮最严重）
1. 坐标语义与现有实现不一致：
	- `tools/analyze_quality.py` 从 Label Studio 读出的 `x/y` 是百分比，再乘 `width/100`、`height/100` 转成像素；
	- `tools/ls_3d_logic.js` 也用 `px = value.x * W / 100`；
	- 这说明项目现实中存在“LS 百分比域”和“分析/3D 消费的像素域”两层表示。
	当前文档直接把内部唯一表示写成“像素坐标 dict”，这会掩盖表示层转换问题，不够专业。
2. 忽略了全景的水平周期性：对 panorama，`x` 方向应视为角度周期，而不是普通图像边界；简单 clamp 会在 seam 附近制造假拓扑。
3. “只改一条边的终点”这种设计过于工程直觉化，没有和附录 A1 的“复现高频失效模式”对应起来。扰动应该按 `operator_id` 决定作用域，而且对 `overextend_adjacent` / 典型 `over_parsing` 这类语义依赖较强的家族，应优先考虑 natural exemplar bank 或专家预批的 surrogate edge，而不是对任意边随机改动。
4. `FailOperator` 允许返回空列表，这在理论上可行，但若写进主实现，需要同步写明：
	- 它属于 intentional invalid；
	- 默认不进入主可比 IoU；
	- 在 userscript / 3D 逻辑 / analyze_quality 下如何被安全消费。
5. 文档大量采用“如果图像尺寸未知，则默认……”的写法，说明接口本身还没锁死。这与“详细执行稿”定位不符。

#### 结论
- 当前版本更像“设计草案 + 解释笔记”，不是最终专业约束；
- 如果要到“每个函数”级别，下一版必须先锁：
  - canonical internal representation（推荐：归一化/百分比域作为冻结清单真源，运行时再映射到像素）
  - 每个 operator 的输入前提、作用域、越界策略、失败码、审计字段
  - `x wrap / y clamp` 的全景规则

## 五、关于“专业要求是否达标”的最终判断

如果标准是“研究生阶段工程实现可落地”，目前这套文档已经够用。

如果标准是“顶刊审稿时，经得住：
- 规则是否与正文一致？
- 是否可复现到函数级？
- 是否会因坐标语义/字段命名/失败处理出现静默漂移？”

那么当前结论是：
- `C2` 基本达到专业要求；
