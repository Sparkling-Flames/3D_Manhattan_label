# 门控失败（Gate Failure）的实际判定流程

根据论文提纲与 analyze_quality.py 数据清洗代码，以下为"某人标注是否产生门控失败"的完整判定逻辑。

---

## 1. 门控失败在论文中的定义（来自提纲 § 2.4.1 标准 Layout 指标与门控）

门控（gate）用于判断某个标注是否可计算标准 layout 指标（2D IoU、3D IoU、深度 RMSE、$\delta_1$）。

### 1.1 两类门控失败

- **可计算型门控失败（Computable-gating failure）**：管道内可自动检测的技术故障
  - 几何归一化失败（normalize_failed）
  - 多边形无法构造或自交（invalid_polygon）
  - 深度图渲染失败（depth_failed）
  - 其他不可恢复的技术错误

- **可比型门控失败（Comparable-gating failure）**：指标可计算但不稳定或不可靠
  - 点对点匹配不稳定（在当前实现中已改为边界 RMSE，降低风险）
  - 中间变量检测异常

### 1.2 关键决策：$(n_{\mathrm{pairs}})$ 不再作为硬门控

- **重要改动**：提纲明确说"n_pairs_mismatch 不再作为标准 layout 的门控条件"
- **理由**：人工修改可能涉及显著的结构改正或大工作量，不应因"改了角点数"而被排除
- **替代**：$\Delta n_{\mathrm{pairs}}$ 作为诊断与分层变量，用于反例筛选和工作量审计

## 2. 实际判定流程（来自 analyze_quality.py :: compute_layout_standard_metrics）

### 2.1 第一步：角点对化（Keypoint Pairing）

输入：标注者提交的原始角点集合（无序）  
过程（_pair_keypoints_to_layout）：
1. 按 x 坐标排序所有角点
2. 自左至右贪心配对：找最近的"兼容" y 值不同的点
   - 兼容条件：$|\Delta x| < \text{threshold}$（默认 5% 图宽）
3. 输出：配对列表（每对包含 $x, y_{\mathrm{ceiling}}, y_{\mathrm{floor}}$）

**可能的第一类失败（奇数个角点）**：
```python
n_points = len(corners)
if n_points % 2 == 1:
    gate_reason = "odd_points"
    return None (计算失败)
```
- **判定**：某人标注了奇数个角点，无法完全配对
- **记录**：`odd_points = True` 在 stats 中
- **影响**：该标注不进入口径 M（可用标准 layout 指标）

### 2.2 第二步：几何归一化（Geometric Normalization）

输入：配对后的角点  
过程（_normalize_to_cor_id_pairs）：
- 转换坐标系统（2D 全景投影 $\to$ 相机空间）
- 检查几何约束（例如天花/地板 y 坐标合理性）

**可能的第二类失败（几何不合理）**：
```python
if normalize_failed(pred_corners) or normalize_failed(ann_corners):
    gate_reason = "normalize_failed"
    return None (计算失败)
```
- **判定**：坐标系转换或几何非法（如投影失败）
- **判定细节**（在 analyze_quality.py 中可扩展）：
  - 点超出全景范围 or 投影无效
  - 边界点配置违反 layout 拓扑规则
- **影响**：该标注不进入口径 M

### 2.3 第三步：覆盖率检查（Coverage Check）

输入：配对后的点数与总点数  
计算：$\text{coverage} = \frac{2 \times n_{\mathrm{pairs}}}{n_{\mathrm{points}}}$（预期 = 1.0，低覆盖意味着有孤立点）

```python
if coverage < min_coverage (default 0.9):  # 90% 
    gate_reason = "low_coverage"
    return None (计算失败)
```
- **判定**：有 >10% 的角点无法配对（孤立点）
- **原因**：可能标注者漏标、误标或人工修改产生了结构异常
- **影响**：该标注不进入口径 M

### 2.4 第四步：多边形合法性检查（Polygon Validity）

输入：配对后的天花/地板 x,y  
过程：
1. 转换坐标系（全景 $\to$ 欧式地面平面）
2. 构造 Shapely Polygon 对象
3. 检验合法性

```python
dt_poly = Polygon(dt_xy)
gt_poly = Polygon(gt_xy)
if (not dt_poly.is_valid) or (not gt_poly.is_valid) or dt_poly.area <= 0 or gt_poly.area <= 0:
    # 无效多边形：自交、退化（零面积）、拓扑错误等
    gate_reason = "invalid_polygon"
    return None (计算失败)
```
- **判定**：多边形自交、退化（顶点共线）、或面积为 0
- **常见原因**：
  - 标注者在全景上重复标注同一个角，导致重复顶点
  - 标注错误导致点序错误而自交
  - 极端情况：所有点在一直线上
- **影响**：该标注不进入口径 M

### 2.5 第五步：深度渲染失败检查（Depth Rendering Failure）

输入：有效多边形的天花/地板点  
过程（_layout_depth_from_cor_id）：
1. 插值边界曲线（通过全景宽度）
2. 按相机模型渲染深度图

```python
gt_depth = _layout_depth_from_cor_id(ann_cor_id, H=height, W=width)
dt_depth = _layout_depth_from_cor_id(pred_cor_id, H=height, W=width)
if gt_depth is None or dt_depth is None:
    # 深度计算失败（三角函数边界情况、NaN、Inf 等）
    gate_reason = "depth_failed"
    return None (用于 depth RMSE/delta_1)
```
- **判定**：深度图包含 NaN、Inf 或 0（无效像素）
- **原因**（在 _layout_depth_from_cor_id 实现）：
  - 某些像素的三角函数在边界处产生数值不稳定
  - 坐标超出有效范围导致 arctan/tan 计算失败
- **影响**：该标注的 depth RMSE 和 $\delta_1$ 无法计算（可记为 NA）
- **注意**：2D/3D IoU 仍可能计算成功

## 3. 门控失败的判定决策树（实际操作）

对于某个标注 $(t, u)$ 的标准 layout 指标计算：

```
开始
  ├─ 角点对化 + 覆盖率检查
  │   ├─ 奇数个角点？
  │   │   └─ YES → gate_reason = "odd_points" → 标记为失败，不计算
  │   └─ 覆盖率 < 90%?
  │       └─ YES → gate_reason = "low_coverage" → 标记为失败，不计算
  │
  ├─ 几何归一化
  │   └─ 投影/几何检查失败？
  │       └─ YES → gate_reason = "normalize_failed" → 标记为失败，不计算
  │
  ├─ 多边形构造 + 合法性
  │   └─ 多边形无效/自交/退化？
  │       └─ YES → gate_reason = "invalid_polygon" → 标记为失败，不计算
  │
  ├─ 深度渲染
  │   └─ 深度图中出现 NaN/Inf/0？
  │       └─ YES → gate_reason = "depth_failed" 
  │           → 2D/3D IoU 可能有效，但 depth RMSE/delta_1 = NA
  │
  └─ 全部通过
      → gate_reason = "" (空) → 计算成功，返回所有指标
```

## 4. 在论文中的报告方式（与提纲对应）

### 4.1 口径划分与门控失败的关系

- **口径 T（Total）**：所有提交的标注（包括门控失败）
  - 报告：总标注数、门控失败数、失败原因分布
- **口径 I（In-scope）**：In-scope 标注（OOS 已剔除）
  - 包括可计算型门控失败（因为仍需记录）
  - 用于相对性指标（如一致性代理）
- **口径 M（Model-usable）**：完全可计算的标注
  - 排除可计算型门控失败 ✓
  - 用于标准 2D/3D IoU、深度 RMSE

### 4.2 门控失败原因分布表（将在 Results 中报告）

建议输出一张表：

| 失败原因 | 数量 | 百分比 | 影响 |
|---------|------|--------|------|
| normalize_failed | - | - | 无法计算 2D/3D IoU、深度 RMSE |
| odd_points | - | - | 无法完全配对，无法计算 2D/3D IoU |
| low_coverage | - | - | 覆盖率 <90%，无法计算 2D/3D IoU |
| invalid_polygon | - | - | 多边形非法，无法计算 2D/3D IoU |
| depth_failed | - | - | 深度渲染失败，无法计算 depth RMSE/δ₁（2D/3D IoU 可能有效） |
| (空) | - | - | 成功计算所有指标 |

### 4.3 Worker 级聚合（Worker × Scene 矩阵中的门控失败率）

对每个 worker u 与场景 s，计算：

$$p_{\mathrm{fail}}(u,s) = \frac{\text{该 worker 在场景 s 中产生可计算型门控失败的任务数}}{\text{该 worker 在场景 s 中的总任务数}}$$

这是异常度 $S_u$ 的五个分量之一，用于工人画像。

## 5. 与提纲其他部分的对应关系

### 5.1 与 $g_t$（结构诊断信号）的关系

- **$g_t$ 的用途**：基于**初始化 $P_t$** 在标注前识别结构高风险
  - 包括"初始化本身产生门控失败"这种模式
- **门控失败率** 的用途：基于**标注者提交 $A_{t,u}$** 在标注后记录失败
- **区别**：$g_t$ 是预测的风险信号；门控失败率是实际的失败计数

### 5.2 与反例筛选的关系

三类反例定义中：
- **Type 3：门控失败**使用此部分的失败原因判定
- 但注意："标注形式上产生了门控失败"≠"标注完全错误"
  - 例如可能只是覆盖率突然降低，而 2D/3D IoU 本身合理
  - 因此反例筛选时需**额外条件**（见 § RQ2 反例筛选）

### 5.3 与可靠度评估的关系

- 门控失败率是 $S_u$（工人异常度）的一个分量
- 低门控失败率 → 工人技术实现稳定
- 高门控失败率 → 可能：工人标注不精确、或倾向于大幅改动结构
